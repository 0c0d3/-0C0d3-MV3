## Getting Started

1. **Fork** the repository on GitHub.
2. **Clone** your fork locally:
   ```bash
   git clone https://github.com/YOUR_USERNAME/0c0d3.git
   cd 0c0d3
   ```
3. Ensure you have **Node.js 18+** and **npm** installed.
4. Install dependencies:
   ```bash
   npm install
   ```

## Project Architecture

### Directory Layout

```
0c0d3/
├── build.js                  # Build script (downloads filter lists, generates rules)
├── package.json              # Node.js dependencies & build scripts
├── BUILD.txt                 # Build instructions (legacy)
├── chrome/                   # Chrome MV3 extension directory
│   ├── manifest.json         # Extension manifest
│   ├── background.js         # Service worker (DNR rule management, state)
│   ├── content.js            # Content script (cosmetic filtering, bridge injection)
│   ├── popup.html / popup.js # Extension popup UI
│   ├── options.html          # Options page
│   ├── inject/
│   │   └── bridge.js         # Injected world script (YouTube blocking, scriptlets)
│   ├── ruleset_*.json        # Static DNR rulesets (generated by build.js)
│   ├── hosts.json            # Host-based blocking rules (bundled data)
│   ├── cosmetics.json        # Cosmetic filter rules (CSS hiding)
│   ├── scriptlets.json       # Scriptlet definitions
│   └── meta.json             # Filter list metadata
└── CONTRIBUTING.md           # This file
```

### High-Level Architecture

| Layer | Technology | Purpose |
|-------|-----------|---------|
| **Build** | `build.js` + `@adguard/agtree` | Downloads 30+ filter lists, parses ABP filters, generates DNR rulesets |
| **Blocking** | `declarativeNetRequest` (DNR) | Static + dynamic rules block network requests at the browser level |
| **Cosmetics** | `content.js` (CSS + MutationObserver) | Hides ad elements and anti-adblock walls via CSS injection |
| **Scriptlets** | `bridge.js` (injected into MAIN world) | Strips ads from YouTube responses (`ytBlock`), modifies fetch/XHR, skips ads |
| **Persistence** | `storage.session` / `storage.local` | Caches filter lists, persists blocked host sets, DNR rules |

### Filter Pipeline

```
Filter Lists (30 sources)
        │
        ▼ build.js (download + parse)
        │
        ├──→ Static DNR rulesets (ruleset_*.json) — up to 150K rules
        ├──→ Host rules (hosts.json) — for dynamic DNR rules
        ├──→ Cosmetic rules (cosmetics.json) — CSS element hiding
        └──→ Scriptlet rules (scriptlets.json) — injected JS behavior
                │
                ▼ Extension loads
                │
                ├──→ background.js loads bundled data
                ├──→ Adds dynamic DNR rules (block + allow)
                └──→ content.js injects bridge.js for scriptlet execution
```

## Development Setup

### Prerequisites

- **Node.js 18+** (required by `@adguard/agtree`)
- **npm** (ships with Node.js)
- A modern browser for testing:
  - **Chrome 120+** (for MV3 `declarativeNetRequestWithHostAccess`)
  - **Firefox 121+** (for MV3 `webRequest` blocking)

### First-Time Setup

```bash
npm install
```

This installs:
- `@adguard/agtree` — ABP filter list parser (AST-based)
- `archiver` — XPI/CRX packaging
- `axios` / `node-fetch` — HTTP downloads
- `sharp` — icon generation

## Building

### Build the Extension

```bash
npm run build
```

Or directly:

```bash
node build.js
```

The build will:
1. Download 30+ filter lists from EasyList, AdGuard, uBlock Origin, and more.
2. Parse all rules using `@adguard/agtree`.
3. Generate static DNR rulesets (`ruleset_0.json` through `ruleset_4_1.json`).
4. Generate bundled host data (`hosts.json`, `hosts_1.json`, ...).
5. Generate cosmetic and scriptlet data files.
6. Package the extension as `0c0d3-3.0.6.xpi`.

> **Note**: The build requires an active internet connection. It completes in 1–3 minutes depending on bandwidth.

### Loading in Chrome (Unpacked)

1. Open `chrome://extensions`.
2. Enable **Developer mode** (toggle top-right).
3. Click **"Load unpacked"**.
4. Select the `chrome/` directory.
5. After making changes, click the refresh icon on the extension card.

### Loading in Firefox (Temporary)

1. Open `about:debugging#/runtime/this-firefox`.
2. Click **"Load Temporary Add-on"**.
3. Select the built `.xpi` file.

For permanent installation, sign the XPI at `addons.mozilla.org`.

## Coding Guidelines

### JavaScript Style

- **ECMAScript**: ES2022+ features are acceptable. The build target is Chrome 120+ and Firefox 121+, both of which support modern JavaScript.
- **Formatting**: Use 2-space indentation. Semicolons are required.
- **Naming**: `camelCase` for variables and functions. `UPPER_CASE` for constants. `PascalCase` for classes.
- **Asynchrony**: Use `async`/`await` consistently. Prefer it over raw Promises or callbacks.
- **Error handling**: Always wrap try-catch around operations that can fail (network calls, Chrome API calls). Log errors via `console.error`.
- **Comments**: Keep them minimal. Explain the *why*, not the *what*.

### Chrome (MV3) Specifics

| Constraint | Rule |
|---|---|
| **Service Worker** | Do not rely on global state surviving SW termination. Persist state via `storage.session` or `storage.local`. Use `alarms` for keepalive. |
| **DNR Rules** | Static rulesets are limited to 330,000 total rules across all enabled rulesets. Dynamic rules are capped at 5,000–30,000 per Chrome version. |
| **Priority** | Allow rules for whitelisted domains **must** have priority > 1000 to override static rules. The project uses `priority: 200000` for dynamic allow rules. |
| **Session Rules** | `updateSessionRules()` is available but rules are cleared on SW termination. Prefer dynamic rules for persistence. |
| **Content Scripts** | Use `"run_at": "document_start"` with `"all_frames": true`. Inject world scripts via `<script>` tags using `web_accessible_resources`. |

### List of Prohibited Practices

- Do **not** use `eval()` or `Function()` constructor.
- Do **not** access arbitrary remote URLs from the content script (enforced by CSP).
- Do **not** hardcode domain lists that belong in filter lists. Use the `WHITELISTED` set only for essential domains that must never be blocked (e.g., `youtube.com`, `googlevideo.com`).
- Do **not** bypass Chrome's DNR limit: always reserve space for allow rules when computing the block rule budget.

## Testing

### Before Submitting

1. **Build cleanly**: Run `npm run build` and confirm it produces zero errors.
2. **Load and verify**: Load the extension in Chrome. Navigate to at least:
   - A major content site (e.g., YouTube, Reddit)
   - A site with heavy advertising (e.g., a news portal)
   - A site with an anti-adblock wall
3. **Check the console**: Open the extension's service worker console (`chrome://extensions` → service worker link) and verify no error messages appear.
4. **Check DNR rules**: Verify that whitelisted domains have corresponding allow rules:
   ```javascript
   // In SW console:
   chrome.declarativeNetRequest.getDynamicRules().then(r => console.log(r.filter(x => x.action.type === 'allow')))
   ```
5. **Verify no regressions**: Ensure that ads are still blocked on standard test pages.

### Manual Testing Checklist

- [ ] Extension loads without errors.
- [ ] Filter lists update successfully (`popup.js` → "Update Lists" button).
- [ ] YouTube video plays without ads.
- [ ] YouTube page load does not show `ERR_BLOCKED_BY_CLIENT` on first-party APIs.
- [ ] Google search results are not broken.
- [ ] Reddit loads without ads.
- [ ] Anti-adblock walls are hidden (test on sites with known anti-adblock scripts).

## Pull Request Process

1. **Create a feature branch** from `main`:
   ```bash
   git checkout -b feat/your-feature-name
   ```

2. **Make your changes** following the [Coding Guidelines](#coding-guidelines).

3. **Test your changes** using the [Testing](#testing) checklist above.

4. **Commit your changes** with a clear, descriptive message:
   ```
   feat: add support for new YouTube ad format
   fix: correct whitelist allow rule priority
   refactor: extract DNR rule builder to separate module
   ```

5. **Push to your fork** and open a Pull Request against the `main` branch.

6. **Ensure the PR description** includes:
   - What the change does and why it's needed.
   - Any relevant issue numbers (e.g., "Fixes #123").
   - Steps for reviewers to test the change.

7. **Address review feedback**. Maintainers may request changes or ask clarifying questions.

### PR Review Criteria

| Aspect | Required |
|---|---|
| Code quality | Follows project conventions, clean variable names, no dead code |
| Performance | No O(n²) loops over large data, respect DNR limits |
| Compatibility | Works on both Chrome 120+ and Firefox 121+ |
| Error handling | Graceful fallbacks, no uncaught exceptions in Chrome API calls |
| Testing | Manual testing evidence or automated test results |

## Reporting Issues

When filing a bug report, please include:

- **Browser** and **version** (Chrome 125, Firefox 128, etc.)
- **Extension version** (from `chrome://extensions` or `about:addons`)
- **Steps to reproduce** (URL, actions taken)
- **Expected behavior** vs **actual behavior**
- **Console errors** (open the extension's service worker console and copy relevant messages)
- **Screenshots** if the issue is visual (cosmetic filtering failures, layout breakage)
